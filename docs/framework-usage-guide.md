# Polisade Orchestrator — Usage Guide

Подробная инструкция для PM, тимлида и инженера, которые хотят использовать Polisade Orchestrator как рабочий фреймворк разработки, а не просто набор slash-команд.

## 1. Что такое Polisade Orchestrator

Polisade Orchestrator превращает CLI-агента в автономную команду разработки с управляемым жизненным циклом:

1. PM формулирует работу естественным языком или через `/pdlc:*`.
2. Фреймворк раскладывает её в артефакты: PRD, SPEC, DESIGN, PLAN, TASK.
3. Агент реализует задачи, пишет тесты, создаёт PR и проводит независимый quality review.
4. PM вмешивается только там, где действительно нужно решение: уточнение требований, снятие блокеров, выбор уровня проработки, ручной merge в режиме `/pdlc:implement`.

Polisade Orchestrator особенно полезен, когда нужно:

- вести разработку через артефакты, а не через хаотичный чат;
- сохранять трассируемость между идеей, спецификацией, задачами и кодом;
- делегировать агенту рутинный цикл `implement → test → PR → review`;
- не терять архитектурный контекст между сессиями.

## 2. Как мыслить фреймворком

Polisade Orchestrator строится вокруг трёх уровней работы.

### Уровень 1. Крупная инициатива

Используйте, если это новый модуль, большая интеграция, эпик на недели или работа с несколькими стейкхолдерами.

```text
/pdlc:prd → /pdlc:spec → /pdlc:design → /pdlc:roadmap → /pdlc:tasks → /pdlc:continue
```

`/pdlc:design` опционален, но практически обязателен для интеграций, API, схем данных, состояний и архитектурных изменений.

### Уровень 2. Обычная фича

Используйте для понятной задачи на несколько часов или дней.

```text
/pdlc:feature → /pdlc:tasks → /pdlc:continue
```

Если по фиче возникает архитектурная неопределённость, добавьте шаг `/pdlc:spec`.

### Уровень 3. Баг, техдолг, хозяйственная задача

Используйте, если работа узкая и понятная.

```text
/pdlc:defect → TASK → /pdlc:continue
/pdlc:debt   → TASK → /pdlc:continue
/pdlc:chore  → TASK → /pdlc:continue
```

## 3. Роли в процессе

### Что делает PM

- формулирует задачу или инициативу;
- отвечает на вопросы, которые агент не должен придумывать сам;
- выбирает глубину проработки, когда Polisade Orchestrator останавливается на checkpoint;
- подтверждает технический контекст для `SPEC` и `DESIGN`, если он не заполнен;
- вручную мержит PR после `/pdlc:implement`, если используется контролируемый режим;
- следит за статусом через `/pdlc:state`.

### Что делает агент

- создаёт и обновляет артефакты;
- декомпозирует работу в атомарные задачи;
- реализует код и тесты;
- запускает регрессию;
- создаёт PR;
- проводит независимый ревью-цикл;
- синхронизирует `.state/PROJECT_STATE.json` с файлами.

## 4. Установка и первый запуск

Полная инструкция по установке уже есть в [README](../README.md), но минимальный сценарий такой:

### Claude Code

```bash
/plugin marketplace add cryndoc/polisade-orchestrator
/plugin install pdlc --scope project
/pdlc:init MyProject
```

### Qwen CLI

```bash
mkdir -p ~/.qwen/extensions
curl -sL https://github.com/cryndoc/polisade-orchestrator/releases/latest/download/pdlc-qwen.zip \
  | bsdtar -xvf - -C ~/.qwen/extensions/
qwen
/pdlc:init MyProject
```

#### Skills vs slash commands в Qwen / GigaCode

Начиная с v2.23.0 бандл для Qwen и GigaCode содержит **два параллельных
пути доступа к основным сценариям** (issue #107):

1. **Slash-команда `/pdlc:<n>`** — явный вызов, как в Claude Code.
   Работают все 26 команд.
2. **Agent Skill** в `skills/pdlc-<n>/SKILL.md` (плоский layout с
   префиксом — Qwen 0.15.1 сканит `<extension>/skills/` без вложенной
   namespace-директории; префикс `pdlc-` развязан с bundled-скиллами
   qwen) — авто-дискаверится агентом по `description`. Если PM пишет
   «закоммить и сделай pr» или «open a PR» обычным текстом, агент
   видит канонический скилл `pdlc-pr` и вызывает `scripts/pdlc_vcs.py`
   вместо импровизированных REST-вызовов.

Оба пути равноправны. Skills эмитятся только для 13 наиболее
«диалоговых» команд (`pr`, `feature`, `defect`, `debt`, `chore`, `prd`,
`spec`, `design`, `roadmap`, `tasks`, `spike`, `review`, `review-pr`);
остальные (`init`, `doctor`, `state`, `implement`, `continue` и т.д.)
остаются slash-only. Allowlist и триггер-фразы живут в
`cli-capabilities.yaml → skills.<name>.{emit_as_skill, intent_triggers}`
(см. [`docs/config-reference.md`](config-reference.md) §
`cli-capabilities.yaml`). Для команд не из allowlist PM просто
использует slash напрямую.

## 5. Что создаёт `/pdlc:init`

После инициализации в проекте появляются:

- `CLAUDE.md` или `QWEN.md` с системными правилами фреймворка;
- `.state/PROJECT_STATE.json` как реестр всех артефактов и статусов;
- `.state/knowledge.json` как накопленный контекст по стеку, паттернам и решениям;
- `docs/` с шаблонами PRD, SPEC, PLAN, TASK, ADR и design package;
- `backlog/` для feature, bug, debt, chore, spike;
- `tasks/` для атомарных задач реализации;
- `.claude/settings.json` для разрешений в Claude Code.

Во время init Polisade Orchestrator также пытается определить стек проекта и предзаполнить `knowledge.json`. Это важно: чем точнее контекст, тем качественнее будут `SPEC`, `DESIGN` и реализация.

## 6. Базовый ежедневный сценарий

Самый частый рабочий цикл PM выглядит так:

1. Создать работу:
   `/pdlc:feature "Нужен экспорт в PDF"`
2. Проверить, что артефакт создан и имеет статус `ready`:
   `/pdlc:state`
3. Разложить фичу на задачи:
   `/pdlc:tasks FEAT-001`
4. Запустить автономную работу:
   `/pdlc:continue`
5. Если агенту нужен ответ PM:
   `/pdlc:unblock`
6. Снова запустить цикл:
   `/pdlc:continue`

Для крупной инициативы перед шагом 3 обычно идут `/pdlc:prd`, `/pdlc:spec`, а часто и `/pdlc:design`.

## 7. Как выбирать правильную команду

| Ситуация | Команда | Когда использовать |
|---|---|---|
| Есть короткое описание новой фичи | `/pdlc:feature` | Основной вход для обычной продуктовой работы |
| Нужен полноценный discovery и формализация | `/pdlc:prd` | Новый модуль, сложный эпик, много неизвестных |
| Нужна техническая проработка | `/pdlc:spec` | API, БД, интеграции, сложная логика |
| Нужны архитектурные артефакты | `/pdlc:design` | C4, ERD, OpenAPI, ADR, glossary, state, deployment |
| Нужно получить план по фазам | `/pdlc:roadmap` | После SPEC для большой инициативы |
| Нужно создать атомарные задачи | `/pdlc:tasks` | После FEAT, SPEC, PLAN или BUG |
| Нужно выполнить одну задачу под контролем PM | `/pdlc:implement TASK-XXX` | Когда нужен управляемый, а не полностью автономный проход |
| Нужно прогнать полный автономный цикл | `/pdlc:continue` | Основная рабочая команда после появления ready TASK |
| Нужно посмотреть, что происходит | `/pdlc:state` | Главная обзорная команда для PM |
| Агент ждёт решений от PM | `/pdlc:unblock` | Для ответов на вопросы и снятия `waiting_pm` |
| Есть баг | `/pdlc:defect` | Репорт дефекта с дальнейшей декомпозицией в TASK |
| Есть техдолг | `/pdlc:debt` | Фиксация инженерного долга |
| Есть небольшая хозяйственная работа | `/pdlc:chore` | Конфиги, обновления, мелкие внутренние изменения |
| Нужен исследовательский заход | `/pdlc:spike` | Когда сначала надо изучить проблему |
| Нужно проверить здоровье проекта | `/pdlc:doctor` | Диагностика состояния и проверок |
| Нужно пересобрать state из файлов | `/pdlc:sync` | После ручных правок артефактов или при рассинхроне |
| Нужно обновить схему state | `/pdlc:migrate` | При апгрейде существующего проекта |
| Нужно проверить расхождение docs и кода | `/pdlc:reconcile-docs` | Advisory-проверка design docs против реализации |

## 8. Подробно про основные рабочие команды

### `/pdlc:feature`

Команда создаёт `Feature Brief` в `backlog/features/`. Это основной вход для большинства запросов.

Что важно знать:

- для маленькой фичи это самый быстрый путь;
- для `M` и `L` размера Polisade Orchestrator делает checkpoint и спрашивает PM, нужна ли спецификация;
- для `L` размера обычно предлагает перейти к `/pdlc:prd`.

Рекомендация: если задача кажется "не очень большой, но есть интеграция или изменение модели данных", почти всегда лучше пройти через `/pdlc:spec`.

### `/pdlc:prd`

Команда запускает интервью и создаёт продуктовый документ, где фиксируются:

- проблема;
- пользователи и JTBD;
- метрики успеха;
- scope и out of scope;
- интеграции и границы ответственности;
- риски, допущения и открытые вопросы.

Используйте PRD, когда проблема ещё не до конца распакована и PM должен сначала договориться сам с собой и со стейкхолдерами.

### `/pdlc:spec`

Команда превращает `PRD` или `FEAT` в техническую спецификацию.

Она нужна, если работа затрагивает:

- API;
- схему БД;
- сложную бизнес-логику;
- внешние интеграции;
- архитектурные решения.

Критически важно:

- `SPEC` опирается на `.state/knowledge.json`;
- если технический контекст пустой, фреймворк обязан остановиться и уточнить стек у пользователя;
- если `PRD` упоминает интеграции, но не фиксирует границы системы, команда переводит работу в `waiting_pm`.

### `/pdlc:design`

Команда создаёт design package в `docs/architecture/` и генерирует только те артефакты, которые действительно нужны по содержанию SPEC или PRD.

Типовые артефакты:

- `c4-context.md`
- `c4-container.md`
- `c4-component.md`
- `sequences.md`
- `data-model.md`
- `api.md`
- `async-api.md`
- `glossary.md`
- `state-machines.md`
- `deployment.md`
- `quality-scenarios.md`
- ADR-файлы в `docs/adr/`

Используйте design package, если хотите оставить после обсуждения не только текст, но и проверяемый архитектурный след.

### `/pdlc:tasks`

Команда раскладывает `PLAN`, `SPEC`, `FEAT` или `BUG` в атомарные задачи в `tasks/`.

Правильное понимание:

- `TASK` в Polisade Orchestrator это основной объект исполнения;
- именно `TASK` должен попадать в `implement` и `continue`;
- для сложных источников команда подтягивает связанный `PRD`, `SPEC`, design package и `knowledge.json`;
- если для SPEC явно нужен design package, а его нет, может сработать design gate и Polisade Orchestrator остановит дальнейшую декомпозицию до решения PM.

### `/pdlc:implement`

Команда реализует одну конкретную задачу. Это контролируемый режим.

Особенности:

- принимает только `TASK-XXX`;
- реализует код, тесты, создаёт PR и запускает review;
- после review не мержит автоматически;
- merge остаётся ответственностью PM;
- если уже есть активная задача в `in_progress`, `review` или `waiting_pm`, повторный `implement` не должен стартовать новую задачу.

Используйте этот режим, когда вам нужен строгий контроль над одной задачей и моментом merge.

### `/pdlc:continue`

Это главный автономный режим. Он сам:

1. выбирает ready-задачу;
2. реализует её;
3. гоняет регрессию;
4. создаёт PR;
5. запускает независимый review;
6. при успешном результате мержит PR;
7. переходит к следующей задаче.

Это команда по умолчанию, когда вы уже прошли стадию формализации и хотите, чтобы агент "просто работал".

### `/pdlc:state`

Главный обзорный экран проекта. Используйте его как диспетчерскую панель.

Он показывает:

- что готово к реализации;
- где можно создать задачи;
- какие артефакты ждут проработки;
- что ждёт решения PM;
- что заблокировано;
- какие design packages и ADR уже существуют.

### `/pdlc:unblock`

Используйте, когда в `state` появляются элементы со статусом `waiting_pm`.

Команда проводит PM через вопросы по одному и после ответа переводит артефакт обратно в `ready`, либо оставляет его в ожидании, либо отменяет.

## 9. Статусы и жизненный цикл артефактов

Базовые статусы, которые вы будете видеть чаще всего:

- `draft` — черновик, ещё не готов к следующему шагу;
- `ready` — можно продолжать pipeline;
- `in_progress` — задача в реализации;
- `review` — PR создан и идёт проверка;
- `waiting_pm` — агент упёрся в решение, которое нельзя домыслить;
- `blocked` — техническая блокировка;
- `done` — работа завершена.

Правильная модель:

- `ready` означает "следующий шаг можно выполнять сейчас";
- `waiting_pm` требует `/pdlc:unblock`, а не очередного `/pdlc:continue`;
- `blocked` означает техническую проблему, а не вопрос к PM;
- `done` для `TASK` ставится только после завершения полного цикла, а не сразу после написания кода.

## 10. Что происходит в автономном цикле

Полный цикл `continue` выглядит так:

```text
IMPLEMENT → REGRESSION → CREATE PR → INDEPENDENT REVIEW → MERGE
```

Это важно по двум причинам:

1. Polisade Orchestrator не считает задачу завершённой просто потому, что код написан.
2. Независимый ревьюер является обязательной частью quality loop.

В типовом процессе:

- код пишет один агент;
- review проводит внешний CLI-ревьюер или текущий агент в режиме `self`;
- при плохом score агент должен исправить замечания и пройти ревью повторно;
- после двух неудачных итераций задача должна уйти в `waiting_pm`, а не быть смерженной "как есть".

## 11. Где хранятся артефакты

### `backlog/`

Здесь живут первичные входы от PM:

- `features/`
- `bugs/`
- `tech-debt/`
- `chores/`
- `spikes/`

### `docs/`

Здесь живут формализованные документы:

- `prd/`
- `specs/`
- `plans/`
- `adr/`
- `architecture/`
- `contracts/`
- `templates/`

### `tasks/`

Здесь живут исполняемые единицы работы.

### `.state/`

Это служебная, но критически важная директория.

- `PROJECT_STATE.json` хранит реестр артефактов, активные списки и настройки;
- `counters.json` выдаёт новые ID;
- `knowledge.json` сохраняет стек, паттерны, команды тестов, решения и словарь.

Практический вывод: редактировать Markdown-артефакты вручную можно, но после этого полезно прогонять `/pdlc:sync`.

## 12. Когда PM должен вмешиваться

Polisade Orchestrator специально не автономен в точках, где "догадка" опаснее паузы.

Типовые случаи вмешательства:

- нужно выбрать `feature` против `PRD`;
- нужно решить, делать ли `SPEC`;
- нужно подтвердить технический контекст проекта;
- нужно уточнить границы системы и контракты интеграции;
- нужно ответить на открытые вопросы;
- нужно принять решение по work item после двух неуспешных review-итераций;
- нужно вручную смержить PR после `/pdlc:implement`.

Если агент задаёт вопрос, это обычно хороший сигнал, а не недостаток автономности: фреймворк защищает проект от ложных предположений.

## 13. Практические рекомендации по использованию

### Начинайте с правильной глубины

- Простая фича: `feature → tasks → continue`
- Сложная фича: `feature → spec → tasks → continue`
- Большая инициатива: `prd → spec → design → roadmap → tasks → continue`

### Держите `knowledge.json` в актуальном состоянии

Если стек, тестовые команды или архитектурные ограничения не зафиксированы, агент будет тратить время на уточнения именно в тот момент, когда вам хочется двигаться вперёд.

### Не пытайтесь исполнять не-TASK артефакты

`FEAT`, `PRD`, `SPEC` и `PLAN` существуют для формализации. Исполнение в Polisade Orchestrator всегда должно сходиться к `TASK`.

### Используйте `state` как главную точку входа

Если не понимаете, что делать дальше, не гадайте. Запускайте `/pdlc:state`.

### Не пропускайте design package там, где есть интеграции

Если в работе появляются API, схемы, состояния, брокеры сообщений или несколько компонентов, отсутствие design package почти всегда приведёт к более слабой декомпозиции и худшему ревью.

## 14. Частые сценарии

### Сценарий 1. Нужно быстро сделать продуктовую доработку

```text
/pdlc:feature "Добавить экспорт в PDF"
/pdlc:tasks FEAT-001
/pdlc:continue
```

### Сценарий 2. Есть большой эпик с новыми интеграциями

```text
/pdlc:prd "Система уведомлений"
/pdlc:spec PRD-001
/pdlc:design SPEC-001
/pdlc:roadmap SPEC-001
/pdlc:tasks PLAN-001
/pdlc:continue
```

### Сценарий 3. Нужен управляемый проход по одной задаче

```text
/pdlc:state
/pdlc:implement TASK-014
```

После этого PM отдельно принимает решение по merge.

### Сценарий 4. Агент остановился и ждёт ответов

```text
/pdlc:state
/pdlc:unblock
/pdlc:continue
```

## 15. Диагностика и обслуживание

Используйте сервисные команды в следующих случаях:

- `/pdlc:doctor` — когда нужно проверить здоровье проекта, конфигурацию, traceability или готовность окружения;
- `/pdlc:sync` — когда state мог разойтись с файлами;
- `/pdlc:migrate` — когда подтягиваете старый проект на новую схему Polisade Orchestrator;
- `/pdlc:questions` — когда нужно отдельно посмотреть открытые вопросы по артефактам;
- `/pdlc:pr` — когда нужны ad-hoc операции с pull request;
- `/pdlc:review` и `/pdlc:review-pr` — когда нужен второй opinion на задачу или на PR.

## 16. Ошибки использования, которых лучше избегать

- Не стартуйте `/pdlc:continue`, если проект завален нерешёнными `waiting_pm` вопросами. Сначала `/pdlc:unblock`.
- Не пропускайте `SPEC`, если работа меняет API, БД или границы системы.
- Не используйте `PRD` для каждой мелкой фичи. Это замедляет поток без пользы.
- Не считайте задачу готовой после одного коммита. В Polisade Orchestrator завершение включает PR и review.
- Не редактируйте `.state/PROJECT_STATE.json` вручную без крайней необходимости. Предпочтительнее обновлять артефакты и затем синхронизировать state.

## 17. Рекомендуемый старт для новой команды

Если вы только внедряете Polisade Orchestrator, используйте такой маршрут:

1. Установите плагин и выполните `/pdlc:init`.
2. Проверьте, что `knowledge.json` корректно описывает проект.
3. Запустите пилот на одной средней фиче через `feature → tasks → continue`.
4. Затем прогоните один более сложный кейс через `feature → spec → design → tasks → continue`.
5. Зафиксируйте внутренние правила команды: когда нужен PRD, когда обязателен design package, кто мержит после `implement`, кто отвечает на unblock-вопросы.

После этого Polisade Orchestrator перестаёт быть "набором команд" и становится воспроизводимым процессом.
